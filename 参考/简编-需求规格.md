# 知著 PenMark — 微信图文编辑器 需求规格

## 项目初衷

做一个 Windows 上用的本地文本编辑器，核心目的是**把微信聊天里的图文内容便捷地收集、整理、导出**。

姊妹产品 **见微 PicMark** 管图片标记，**知著 PenMark** 管文字写作——"见微知著"，一图一文。

使用场景：微信里看到有价值的聊天记录（文字+图片混排），想快速整理成一篇文章保存下来。操作要尽量少——选中聊天内容复制粘贴，图片直接拖进去，编辑排版，一键导出。

不需要登录，不需要注册，一个文件双击就能用。

**关于联网**：本地编辑体验必须又快又稳（离线优先，核心操作绝对不能依赖网络）。联网功能是加分项——比如事后同步到云端、发布到某处、用 AI 辅助处理内容等。参考 WPS 的模式：离线编辑丝滑，云同步是可选的辅助。总之，"快"是底线。

---

## 核心功能

### 1. 从微信导入内容

**拖入图片**：在微信聊天窗口里按住图片，直接拖到编辑区域，图片自动插入到光标位置。支持一次拖多张。

**粘贴图片**：微信里右键"复制"某张图片，回到编辑器 Ctrl+V 粘贴。剪贴板里有图片就会插入。

**粘贴文字**：微信里选中一段文字 Ctrl+C，回到编辑器 Ctrl+V 粘贴，保留换行。

**混合粘贴**：如果剪贴板里同时有文字和图片（比如从某些富文本源复制），两者都保留，不丢内容。

### 2. 富文本编辑

所见即所得的可编辑区域，输入文字、调整格式和排版。

支持的基本格式：
- **文字样式**：加粗、斜体、下划线、删除线
- **段落样式**：正文、标题1/2/3、引用块（blockquote）
- **列表**：有序列表、无序列表
- **分隔线**
- **清除格式**

### 3. 图片管理

插入编辑器中的图片需要能够：
- **缩放**：选中图片后，四角出现拖拽手柄，拖动等比缩放
- **拖拽移动位置**：按住图片可以直接在文档中拖动到其他位置
- **删除**：选中图片后按 Delete 或 Backspace 删除，且删除操作可撤销

图片应该看起来是文档的一部分，而不是浮动的画布层。

### 4. 撤销/重做

所有操作——文字输入、格式调整、图片插入、图片移动、图片删除——都应该支持 Ctrl+Z 撤销和 Ctrl+Y 重做。

### 5. 导出

至少支持三种导出格式：

- **HTML**：自包含的网页文件，图片以 base64 嵌入，一个文件包含全部内容，浏览器直接打开
- **Markdown**：标准 `.md` 文件，标题/列表/引用/图片都正确转换
- **Word (.doc)**：图片 base64 嵌入的 HTML 格式 `.doc`，Word / WPS 可直接打开编辑，**图片不能丢失**（这是最基本要求）

### 6. 文件管理

- **保存**：导出为 HTML 文件（Ctrl+S），下次可以重新打开继续编辑
- **打开**：打开之前保存的 HTML 文件（Ctrl+O），恢复编辑状态
- **新建**：新建空白文档（Ctrl+N）
- **自动恢复**：意外关闭浏览器后，下次打开能恢复上次编辑的内容

## 用户体验要求

### 极简设计

**审美参考：Claude 的排版风格**——温润、克制、有呼吸感。

- 底色偏暖白 / 米白（不是纯 #FFF 的刺眼白），类似纸张质感
- 字体偏好衬线或高质量系统字体，中文用有质感的宋体/楷体类或优雅的无衬线
- 行间距宽裕，段落间距分明，文字不挤
- 没有任何高饱和颜色、渐变、花哨图标、红绿徽章
- 工具栏简约，分组清楚，图标用极简线条或纯文字
- 整体气质是"写字的地方"而不是"操作面板"

### 操作流畅

- 一个 HTML 文件，双击即用，不需要安装
- 主要操作用键盘快捷键完成
- 拖入图片时有视觉反馈（拖入覆盖层提示）
- 操作结果有轻量提示（底部 toast 通知）

### Windows 平台特性

- 在 Windows 上使用体验要好（默认浏览器打开即可）
- 状态栏显示字数统计和图片数量
- 可附带一个 Python 启动脚本（.pyw，不弹黑窗口）或用其他方式模拟桌面应用体验

---

## 边界条件

- 所有图片数据以 base64 嵌入，文档完全自包含，不依赖外部文件
- 微信粘贴的图片要注意处理剪贴板中的 CF_DIB 格式
- 微信拖出的图片可能是临时文件路径，要正确处理
- 编辑器内的图片是文档内联元素而非浮动层
- 删除操作也要进入撤销栈
- 加载旧文档时保留原有格式（图片支持缩放、拖拽、删除）

---

## 不做的事（核心版）

- 不需要注册登录才能用
- 不需要复杂的权限管理
- 不需要实时协作
- 不需要 Markdown 源码编辑模式
- 不需要移动端适配（桌面端专属）

## 加分项（有最好，没有也行）

- 云端同步 / 多设备备份（事后同步，不拖慢编辑速度）
- AI 辅助写作 / 润色 / 总结
- 直接导出到在线文档平台（腾讯文档等）
- 打包成 exe 安装包（比纯 HTML 文件更像桌面应用）

---

## 技术偏好（仅供参考）

- 用户倾向 Windows 原生方案（WPF / .NET Framework 4.7.2），但不强制
- 如果用 Web 方案，单文件 HTML（CSS + JS 内联）即可
- 纯前端，无服务器依赖
- 第三个选项：用 Tauri / Electron / WebView2 打包成 exe 给更好的桌面体验（加分项）

## 版本演进中的经验教训

以下是开发过程中踩过的坑，供参考：

1. **图片缩放**：如果依赖 CSS `resize` 属性，在 contenteditable 内的 `contenteditable="false"` 容器上可能不生效，需要用自定义拖拽手柄
2. **撤销支持**：直接操作 DOM（`insertNode`、`remove`）不进入浏览器 undo 栈，要用 `execCommand('insertHTML')` 这类命令式 API
3. **图片拖拽移动**：需要区分"编辑器内图片拖拽移位"和"从外部拖文件进来插入"，两者共用 document 级 drag/drop 事件时容易冲突
4. **Word 导出**：用 HTML 套 `msword` MIME 类型的假 .doc 是最可靠的方案（图片 base64 嵌入绝不丢）；真实 OOXML .docx 需要处理 ZIP 打包、OOXML 命名空间、图片提取等，容易出问题
5. **图片容器**：图片用 `contenteditable="false"` 包裹可以防止用户在图片上误编辑，但也要设 `draggable="true"` 支持拖拽
6. **粘贴混合内容**：剪贴板同时有文字和图片时，优先处理图片再追加文字，不要只取其一
7. **自动保存**：用 localStorage 做自动保存，设置合理间隔（5秒），注意处理存储溢出的情况
