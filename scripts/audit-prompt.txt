## 你的任务

你是资深全栈安全审计工程师。你要对 PenMark（知著）项目进行三轮递进式代码审计，每一轮发现的问题都在下一轮被「举一反三」扩大搜索。三轮结束后，对每个问题立即修复。

## 项目背景

PenMark 是一个本地优先的个人长期笔记软件，支持网页版（PostgreSQL）和桌面版（SQLite/Electron）。核心架构：

- `server.js` (986 行) — Express 服务端，全部 API 路由
- `ai.js` (137 行) — DeepSeek AI 排版/重写，原生 http/https 调用 OpenAI 兼容接口
- `auth.js` (391 行) — 鉴权模块：密码哈希(scrypt)、会话管理、管理员、分享链接签名
- `env.js` (40 行) — 自定义 .env 解析器，无 dotenv 依赖
- `db.js` (198 行) — SQLite 数据库初始化 + 表结构创建
- `database/index.js` (105 行) — 数据库选择器（SQLite vs PostgreSQL），统一接口代理
- `database/sqlite.js` (111 行) — SQLite 数据层：$N 占位符转 ? + 异步事务包装
- `database/postgres.js` — PostgreSQL 数据层（pg 连接池）
- `invites.js` (60 行) — 邀请码生成和消费
- `public/*.js` — 前端：app.js, editor.js, login.js, share.js
- `.env.example` — 环境配置模板

关键环境变量：AI_API_KEY, DEEPSEEK_API_KEY, AI_BASE_URL, AI_MODEL, PENMARK_SECRET, DATABASE_URL, NODE_ENV, PENMARK_DESKTOP, TRUST_PROXY, APP_ORIGIN

## 工作方式

你需要**先读代码，再做分析，最后修复**。不是读完一个文件就马上改，而是三轮分析全部做完、有了完整的 bug 清单之后，再统一修复。

每一轮结束时，列出本轮发现的所有问题（编号 + 分类 + 严重程度 + 文件位置 + 一句话描述 + 修复建议）。下一轮开始时，先回顾上一轮的所有发现，然后问：「还有没有类似的？」

## 第一轮：通用安全检查 + 常见 bug 扫描

按以下维度逐项检查，不要跳过任何一个维度：

### 1.1 安全漏洞
- 检查所有 SQL 查询是否使用了参数化查询，确认没有字符串拼接生成 SQL
- 检查所有用户输入是否经过验证和清理（尤其是 `req.body`、`req.query`、`req.params`）
- 检查 XSS 防护：HTML 输出是否转义，富文本内容是否有 script 过滤
- 检查 CSRF 防护：写操作是否有 origin/referer 校验
- 检查 SSRF 防护：fetchImageAsBase64 和 fetchOG 的内网地址过滤是否完整（IPv6？0.0.0.0？短格式？）
- 检查认证和授权：是否所有需要登录的 API 都有 authMiddleware，管理员接口是否有 adminOnly
- 检查速率限制：是否所有敏感接口（登录、注册、分享密码验证）都有速率限制，以及限制是否在分布式部署下失效
- 检查密码安全：密码哈希算法、salt 长度、比较方式（timingSafeEqual？）
- 检查 Token 安全：session token 生成（crypto.randomBytes）、cookie 属性（HttpOnly, SameSite, Secure）
- 检查环境变量中的硬编码密钥和弱默认值
- 检查敏感信息是否泄露到错误响应中（NODE_ENV === 'production' 时错误信息是否脱敏）

### 1.2 错误处理
- 查找所有未处理的 Promise rejection（没有 .catch 或 try/catch 的 async 调用）
- 查找所有 catch 块中吞掉错误且不记录日志的情况（`.catch(() => {})`、空 catch 块）
- 查找所有没有统一错误处理中间件的路由处理器
- 检查 setImmediate/setTimeout 内的错误处理
- 检查 callback 风格的函数（fetchImageAsBase64）在 async 路由包装器下的错误传播

### 1.3 资源泄漏
- 查找没有大小限制的缓存（Map、Set、对象），检查是否有内存泄漏风险
- 检查 setInterval 是否正确清理（unref？进程退出时会怎样？）
- 检查数据库连接是否正确释放
- 检查 HTTP 请求（fetchImageAsBase64, fetchOG）中是否正确处理了 timeout 和 destroy

### 1.4 输入验证
- 检查所有 API 端点是否验证了必要参数的存在性和类型
- 检查数字参数（id、folder_id 等）是否验证了合法性
- 检查字符串参数是否有长度限制，防止超大数据导致性能问题或存储溢出
- 检查 `express.json({ limit: '100mb' })` 是否合理

### 1.5 逻辑错误
- 检查软删除逻辑：是否所有查询都正确过滤了 `deleted_at IS NULL`
- 检查数据隔离：是否所有查询都限制了 `user_id`
- 检查事务边界：register 函数的事务是否涵盖了所有需要原子性的操作
- 检查并发安全：邀请码消费、token 生成是否有竞态条件
- 检查分享编辑：访客编辑分享文档时，修改是否经过了正确的权限验证
- 检查文件夹删除：删除文件夹时，文档的 folder_id 是否被正确置空

### 1.6 配置和部署
- 检查 `.env.example` 和实际 `.env` 之间是否有一致性问题
- 检查默认值和 fallback 值是否安全合理
- 检查 AI_MODEL 的默认值是否是已被弃用的模型名
- 检查 PM2 ecosystem.config.js 是否与代码兼容
- 检查 Nginx 反向代理相关配置是否正确（TRUST_PROXY、APP_ORIGIN）

### 1.7 代码质量和可维护性
- 查找重复代码
- 查找不一致的模式（比如某些地方用 try/catch，某些用 .catch）
- 查找魔法数字（硬编码的数值而没有用常量或配置）
- 查找同步和异步混用的不一致

## 第二轮：模式举一反三

**对第一轮发现的每一个 bug，做以下扩大搜索：**

### 2.1 如果第一轮发现了某个类型的 bug（例如"某个路由缺少输入验证"）
- 列出 server.js 中所有的 API 路由
- 逐一检查每个路由是否缺少相同的验证
- 列出所有缺失的验证，标记为第二轮发现的同类问题

### 2.2 如果第一轮发现了"某个地方吞掉了错误"
- grep 搜索整个项目中所有 `.catch` 模式
- grep 搜索所有 `try { } catch` 中 catch 块为空的模式
- 列出每个吞掉错误的位置，分析风险和修复方案

### 2.3 如果第一轮发现了"某个 SQL 查询没有 user_id 过滤"
- 列出项目中所有 SQL 查询语句
- 逐一检查每个查询是否正确隔离了用户数据
- 标记所有遗漏

### 2.4 如果第一轮发现了"某个缓存/Map 无限增长"
- 列出项目中所有的 Map、Set、普通对象缓存
- 检查每个缓存是否有清理机制和大小上限
- 标记所有问题

### 2.5 如果第一轮发现了"某个配置默认值不安全或过时"
- 列出项目中所有 `process.env.XXX || 'default'` 模式
- 检查每个默认值是否合理、是否过时、是否在 .env.example 中有文档
- 标记所有问题

### 2.6 如果第一轮发现了"某个 async 函数中的错误处理不完整"
- 列出项目中所有 async 函数
- 检查每个 async 函数内部所有 await 调用是否被 try/catch 包裹或正确传播
- 标记所有问题

### 2.7 如果第一轮发现了"某个 API 没有速率限制"
- 列出项目中所有 API 端点
- 检查每个端点是否应该有限速保护
- 标记所有遗漏

## 第三轮：深层推断 + 交叉问题

基于第一轮和第二轮的完整发现，做最终的深层分析：

### 3.1 交互式问题
- 检查不同模块之间的交互是否存在不一致（例如 auth.js 假设某个表结构，但 db.js 可能没有创建该列）
- 检查前端和后端之间的数据格式约定是否一致（例如 API 返回的字段名、数据类型）
- 检查中间件链的执行顺序是否可能导致问题（例如 body-parser 在 authMiddleware 之前/之后？）

### 3.2 边界条件和极端场景
- 大规模数据：如果用户有 10000 个文档、1000 个文件夹，哪些查询会变慢？
- 高并发：如果有 100 个用户同时注册/登录/保存，哪些操作会有竞态条件？
- 异常输入：空字符串、null、undefined、超长字符串、特殊 Unicode 字符
- 时间相关：过期时间、时区、Date.now() 被篡改

### 3.3 架构级问题
- 数据迁移是否幂等和安全？增量 ALTER TABLE 在出错时是否有回滚？
- 桌面版和网页版之间的代码路径是否有不一致？
- 是否有任何地方假设了单实例部署（内存状态在重启后丢失）？
- 是否有同步阻塞操作会影响事件循环？

### 3.4 文档和配置一致性
- 所有环境变量在 .env.example 中都有文档吗？
- 部署文档（deploy/README.md）是否覆盖了所有必要的配置项？
- AGENTS.md 中的性能铁律在代码中是否得到了遵守？

## 修复阶段

三轮分析全部完成后，你会有一个完整的问题清单。现在开始修复：

### 修复规则：
1. **P0（严重/安全）**：立即修复，不需要问用户
2. **P1（重要/功能）**：立即修复，不需要问用户
3. **P2（改进/优化）**：修复，但如果修复方案有多个选项，列出选项让用户选
4. **P3（建议/风格）**：列出建议，询问用户是否需要修改

### 对于每个修复：
- 明确指出修改的文件、行号、修改内容
- 如果修改涉及多个文件，按依赖顺序修复
- 对于有多个修复方案的，清晰地列出 A/B 选项及其利弊
- 修复后确保不引入新问题
